# Memory - learned patterns (append across runs)

Durable lessons so the skill compounds across client projects rather than relearning. The orchestrator reads this at Stage 0 and appends at Stage 7. Distil periodically with `anthropic-skills:consolidate-memory`.

## Brochure parsing
- CBRE CEE brochures come in two layouts: a numbered TOC on page 1 (Pilsen set), or no TOC where page 1 is already a property (Budapest/Bratislava sets). `extract_pdf._is_toc_page` distinguishes them.
- Labels can wrap across lines ("Permitting status and\ntimeline") - the label regex uses `\s+` between words.
- "Warehouse" (rent) vs "Warehouse Area" (size), "Office" vs "Office Area" - match labels on their own line, longest-first, so the shorter never shadows the longer.
- Rent values are prefixed with junk ("(10m clear height): EUR 62,4...") - take the number after the euro sign.
- Area values are prefixed ("BOR 10: 35 604 sq. m.") - take the largest plausible number.
- Motorway is in prose, not a label - mined from the description.

## Numbers
- European format throughout: space/NBSP/thin-space thousands, comma decimals (`62,4` = 62.4; `39 471` = 39471). `normalize.normalize_number` centralises this; build the golden test from real brochures.

## Geocoding
- HU/SK brochures omit lat/lng; Nominatim fills them (1.1s rate limit, cached in `reference/geocode_cache.json`). Compound city strings ("Triblavina, Greater Bratislava Area") do not resolve - either add a `city_country`/centroid entry or accept the gap.

## Images
- PDF embedded rasters reliably give 30/30 heroes; PPTX `MSO_SHAPE_TYPE.PICTURE` sometimes yields none (images in groups/backgrounds) - PDF is the dependable image source.

## Developer aliases seen
- CTPark = CTP; Panattoni Park = Panattoni; VGP Park = VGP; P3 Logistic Parks = P3. Extend `match.DEV_ALIASES` as new ones appear.

## Template
- The raw reference HTML assumed complete data; v2 adds a `fmt()` robustness patch so a missing numeric shows "tbd" instead of crashing the grid. v3 adds guards on the comparison-table distance rows (`groupedDistances(p).<type>[0]` is undefined when a POI type is empty or POIs are off) so the compare view does not crash. Any future chrome edit -> bump the version.

## QA hardening (post-build review, 2026-06-05)
- **Injection safety:** data is serialised with `ensure_ascii=True` and angle brackets escaped to their `\uXXXX` unicode escapes (`<`/`>`) in `build_dashboard.block()`, so source-derived text containing `</script>` or U+2028/U+2029 cannot break out of the `<script>` block. `validate-html` asserts the delivered file carries exactly the template's `<script>` tag count. (Build the escape with `chr(92)`, never a literal backslash - the double-escaping is easy to get wrong.)
- **Determinism:** `merge_cluster` iterates `sorted(fields)` and `block()` uses `sort_keys=True`, so the same inputs produce a byte-identical `built.html` across runs/machines (verified). `meta.generatedAt` differs per run but is not rendered.
- **Honesty is mechanical, not just reviewer-dependent:** `gate_runner.py trace-coverage` blocks any populated field with no ledger row (catches a fabricated value added to a field that had none); G-trace (reviewer) still verifies value-vs-source for fields that DO have rows. `merge.py` now emits a ledger row for every `tbd` gap and records discarded conflict values into `conflict_note` + `meta.conflicts`.
- **`run.py` stops before build if `validate-data` blocks** (no more empty-dashboard-with-exit-0). `final_gate.py` requires the four judgement verdict files (FAIL if missing) unless `--no-reviews` acknowledges DEGRADED mode; verdict detection is a tolerant regex (`**VERDICT:** red`, `VERDICT - red`, etc. all block).
- **normalize_number:** ranges ("10-12") return None (no fabricated lower bound); multi-dot EU thousands ("1.234.567") parse.
- **match.same_property:** same-name buildings across sources with warehouse area differing >15% are kept distinct (no silent loss of a phase).

## Speed (quality-preserving, 2026-06-05)
- **Parallel reviewers:** the pre-build judgement gates (G-honesty, G-trace, G-images) are dispatched as ONE concurrent batch of separate `Agent` calls (not sequential, never one merged agent); G-visual runs post-build. Concurrency strengthens independence (no verdict exists yet to bias another) and cuts that stage's wall-clock to the slowest single review.
- **Freeze guarantees parallel == quality:** `gate_runner.py freeze <canonical>` snapshots the bytes before dispatch; `freeze --check` after (and inside `final_gate.py`) proves every concurrent reviewer judged byte-identical bytes and no edit slipped in mid-review. This is what makes parallelism provably quality-neutral.
- **Model tiering:** G-honesty + G-trace = Opus/High (fabrication is the worst failure); G-images + G-visual = Sonnet (image-match/layout are easier, faster). All four gates + fresh-context independence retained.

## Live nearest-POI discovery, server + client (v5 chrome, 2026-06-05)
- **Principle (broker-set, non-negotiable):** the dashboard shows the GENUINE nearest port/airport/rail/border/city per property, discovered dynamically - NEVER a preloaded stand-in. A curated POI set was rejected because the skill would lazily return the nearest *of that set* (a far major port) instead of the true nearest (a smaller local one) - e.g. a Barcelona dry port on a Malaga longlist. So `assets/poi_library.json` is RETIRED from the nearest path (it still seeds the geocode city-centroid offline fallback, intake's city->country clustering, and merge's offline POI seed - and when Overpass is unreachable, enrich now KEEPS those merge-seeded POIs instead of wiping them).
- **Server-side (`enrich.attach_pois(canonical, gaps)`): PRIMARY = the bundled complete dataset** (`assets/poi_dataset.json.gz`, ~6,900 facilities: all scheduled airports via OurAirports public domain, all ports incl. terminals - operators/offices filtered out by itemtype, all SGKV intermodal terminals; built by `helpers/build_poi_dataset.py` from org exports, auto-detected schemas). Nearest air/port/rail = pure offline computation; "nearest of a complete set" IS the genuine nearest (verified: Puerto Seco Azuqueca at 0.9 km vs 24.5 km via OSM discovery, Sagunto as true nearest seaport). Borders/cities from the curated library. The OSM path below is the FALLBACK when the dataset file is absent: live nearest-of-type, ONE combined query at the calibrated 180 km radius (empirically: 180 km = 31KB and all types; 540 km+ OOMs the public server at 2 GB, even nodes-only, on the main server AND mirrors - do NOT re-add big-radius or expanding scans). Output is split (`out qt` for nodes, `out tags center qt` for ways) so polygons ship without node-ID arrays; ways are REQUIRED (major airports/ports are polygons - the IATA tag sits on the perimeter). Beyond 180 km, missing types are supplemented from the curated major-facilities library, capped by `POI_MAX_KM` and LABELLED "library - beyond the 180 km OSM scan" in the note. Freight-only classification: marinas (`leisure=marina` / marina seamark category) and airport security checkpoints (`amenity=security_control`, indoor `level`) are excluded - both shipped as junk POIs on a real Madrid run. An HTTP-200 body with a `remark` runtime error is a FAILURE (raise/skip), never "no POIs nearby" - caching it poisons the cache silently. Circuit-breaker after the first failure; 2 s politeness gap between queries.
- **Client-side (template v5):** `ensureNearestPois`/`rebuildPoiMarkers`/`populateMapPois` + hooks in `openModal`, `openCompare`, `switchView('map')`. On modal/compare/map-view open it queries Overpass live (the broker's browser has real internet, unlike the build sandbox), pushes the true-nearest into `POIS` (const array, mutated via `.push`) and refreshes markers/table. Offline/blocked/CORS -> silent no-op (keeps baked POIs + honest tbd), NO console error. Overpass sends `Access-Control-Allow-Origin: *`, so the browser fetch works.
- **Sandbox reality:** Overpass is BLOCKED from this build sandbox (proxy), though OSRM/Nominatim worked - so the **client-side path is the PRIMARY one** that actually delivers "always nearest"; server-side is best-effort and degrades to gaps.
- **v5 chrome was HAND-APPLIED** to `assets/dashboard_template.html` (raw reference not on disk); VERSION -> v5 + new `chrome_sha256`. POIS stays `const` so no `build_dashboard`/gate-regex change was needed. If regenerating from a raw reference, re-add the v5 block (see `make_template.py` v5 NOTE).
- **Validated offline:** `node --check` PASS (full app JS parses), `validate-html` byte-identical to render with the new chrome SHA, smoke PASS, server-side degrades to honest gaps. **NOT validated:** live Overpass query QUALITY + in-browser runtime (Overpass blocked here) - confirm in a networked browser (open a Malaga option -> nearest port should resolve to Malaga/Algeciras, not Barcelona; the Overpass tag filters are tunable if a market returns a wrong feature).
- **Modal district/region guard (v6, FIXED 2026-06-05):** `detailHTML` previously dereferenced `p.district.*` and `REGIONS[p.regionCode]` without guards, so opening a detail modal on a build with no district/regions threw a TypeError and the whole modal broke (and nothing in the pipeline auto-populates `district`). v6 guards it: `const reg = REGIONS[p.regionCode] || null; const dist = p.district || null;` + `rNum`/`rK`/`rStr` helpers; the Workforce & Region block renders the district-panel only when `dist` exists and the region grid only when `reg` exists, with partial figures shown as `tbd`, else an honest "not yet added - see Gaps Report" placeholder. Runtime-verified in node (no throw on no-district/no-region or partial-region); `validate-html` byte-stable at v6. All `p.district`/`reg.`/`REGIONS[` derefs were confined to this one detailHTML section.

## Multilingual extraction + rapidfuzz fallback (works across Europe, 2026-06-05)
- **Root cause of the slow TEDi (ES) run:** `extract_pdf.py` was English-only (LABELS + the page-gate keyed on "Owner/developer"/"Property Status"/"Warehouse Area"), so Spanish decks matched ~0 labels -> 0 records -> all-manual authoring, which itself seeded a fabricated-gap defect. Fixed.
- **Data-driven label ledger:** `assets/label_ledger.json` maps each canonical field to multilingual aliases (EN/ES/FR/DE/IT/NL/PT/PL). `extract_pdf._load_ledger_labels()` appends them to the in-code English LABELS (English stays the baseline -> zero CEE regression), de-duped on (phrase, field). This is now the PRIMARY extension point - extend it per market and bump its `version`.
- **Language-agnostic page-gate:** a page is a property page when `len(_find_labels(text)) >= 2` (any language), not on English anchor strings. `_is_toc_page` and `extract_pptx` use the same rule.
- **Monthly->annual rent:** `MONTHLY_MARKERS` (/mes, /mois, /Monat, /mese, /maand, ...) trigger x12 to the dashboard's "/ year" basis; the conversion is written into the ledger locator (e.g. "page 3 (4,75/m2/mo x12)") - honest, not hidden. Verified 4,75 EUR/m2/mes -> 57.0.
- **Negative coords:** `LATLNG` now allows a leading minus, so western/southern (ES/PT/FR/UK) longitudes parse with the correct sign (was silently placing markers in the eastern hemisphere).
- **Multilingual unknowns:** `normalize.looks_unknown` now treats "a consultar"/"sur demande"/"auf Anfrage"/"su richiesta"/"op aanvraag"/... as tbd.
- **rapidfuzz fallback:** `helpers/rapidfuzz_shim.py` (difflib-backed ratio/token_set_ratio/token_sort_ratio/process) + try/except guards in `match.py` and `extract_xlsx.py`, mirroring the fitz shim. jsonschema was already optional, so all three of the TEDi run's hand-written shims (fitz, rapidfuzz, jsonschema) are now skill-native; the deterministic spine runs offline with no pip. Dedup quality still policed by the coverage gate.
- **Validated** (synthetic EN+ES deck vs real fitz): ES fields incl. x12 rent and negative lng correct; EN regression identical; rapidfuzz-absent dedup still merges cross-source; smoke byte-stable. **NOT yet validated on a real TEDi brochure**, and the own-line label parser does NOT handle inline "Label: value" decks (those need the vision/LLM fallback). Drop a real ES brochure to validate/tune before production trust.

## fitz shim for locked-down sandboxes (no pip/network, 2026-06-05)
- The Cowork bash sandbox can block pip AND outbound network, so `PyMuPDF` (`fitz`) and `pillow-heif` may be uninstallable. `helpers/fitz_shim.py` is a drop-in stand-in for the nine `fitz` calls the skill uses (`open`, `doc[i]`, `page_count`, `extract_image`, `close`, `Page.get_text`, `Page.get_images`, `Page.get_pixmap`, `Pixmap.tobytes`), backed by **pypdfium2** (page render + embedded-image extraction) and **pdfplumber** (page text). `extract_pdf.py` and `images.py` do `try: import fitz / except: import fitz_shim as fitz` (with `sys.path` set first), so the real library is always used where present and never shadowed; the shim is purely the no-fitz safety net. Only those two files import fitz; `extract_pptx.py` reaches it only via `images.py`.
- **Text backend = pdfplumber, NOT pypdfium2.** A/B test vs real fitz on a structured brochure page (TOC + label-anchored spec pages): pdfplumber's `extract_text()` reproduced the parser output byte-identically (`BOR 10: 35 604`->35604, `BTS up to 60 000`->60000, `€ 4,75`->4.75, lat/lng, `D5` motorway from prose, full description); pypdfium2's `get_text_range()` lost the per-line label structure and extracted nothing. `fitz_shim.TEXT_BACKEND` ("pdfplumber"|"pdfium") switches it if ever needed.
- **Validated:** image extraction/render matches fitz exactly (same embedded dims (820,520), byte-identical 3839-byte JPEG hero). Field extraction identical on a synthetic single-column page. NOT yet validated on a real multi-column/table brochure - pdfplumber reconstructs text spatially, so a real CBRE brochure with columns/tables could order text differently from fitz and break the label parser; validate field extraction on a real sample before trusting the shim in production (diff shim-vs-fitz on a brochure PDF), or just rely on real PyMuPDF wherever it is installed.
- `pillow-heif` is already optional (try/except in `images.py`); its absence only drops HEIC support, nothing else.

## Performance (in-process spine + parallel OSRM, quality-preserving, 2026-06-05)
- **In-process spine:** `run.py` and `final_gate.py` import the helper modules once and call their `main()` in-process instead of spawning a fresh `python` per stage/gate. The heavy libs (fitz/Pillow/rapidfuzz) load once, not ~14x, removing several seconds of fixed start-up. Behaviour/output identical; each stage is wrapped in try/except so one failure is reported and the run continues (old subprocess isolation preserved). `run.py` imports the helpers inside `main()` (not at module load) so it still imports cleanly where an optional reader's dep is missing; the three extractors degrade to a skip if unavailable. NOTE the work-dir CSV variable is `ledger_csv` now (it would otherwise shadow the imported `ledger` module).
- **Parallel OSRM:** `enrich.osrm_prebake` fetches the (property x POI) routes through a bounded `ThreadPoolExecutor` (`OSRM_WORKERS=6`) with a 429-backoff retry, instead of a serial loop with `time.sleep(0.2)`. Same routes, same km/min values, no POI dropped - the deep-sea gateways (e.g. Hamburg ~676 km from Pilsen) are deliberately KEPT, NO distance cut - so it is a pure latency win (~8x at scale). The worker cap keeps the shared public OSRM server polite.
- **PDF opened once for images:** `images.hero_for_pdf_page` reuses a cached open `fitz.Document` per path (`_get_doc` / `close_doc_cache`, the latter called at the end of `merge.main`) instead of reopening the brochure once per property. Identical image bytes.
- **final_gate gate set trimmed (no quality loss):** drops the redundant `self-check` (chrome SHA is re-checked by `validate-html`) and, in the reviewed path, the redundant `validate-data` (the `freeze --check` proves canonical is byte-identical to the bytes pre-build validate-data already cleared). DEGRADED `--no-reviews` mode still runs `validate-data` as the schema net (no freeze proof there). `validate-html` (byte-identity) + `reconcile` always run; all verdict files still required (G-enrich too when regions ran). None of these touch the template (the chrome version lives in `assets/VERSION` - currently v10; trust the file, not this note).

## Hero image selection (real CBRE ES decks - TEDi run)
- A card hero MUST be the page's real PHOTO / AERIAL / RENDER, never a road MAP, a flat PLAN diagram, or a "screenshot of the slide" (whole-page render). The colourfulness-only `photographic_score` mis-ranked these on real decks: a Google-Maps screenshot scored 18.85 while a real (near-monochrome) warehouse interior scored 3.66 < MODEST_PHOTO and was DROPPED, so the map/plan/page-render led the card.
- Fix = `images.classify_image`: CONTINUOUS TONE (luminance entropy + low flat-fill share), NOT colour, separates photo/aerial/render (lumEnt >= ~6.6) from plan (~6.0) / road map (~4.3, pale desaturated palette) / icon (single dominant colour). `HERO_TIER` ranks photo>plan>map>text>logo; `page_hero_and_plan` Tier A and `_deck_photo_index` pick by TIER (a photo leads even when a map scores higher) and a PHOTO-kind image is a hero even below MODEST_PHOTO. Plans/maps still go in the gallery + the Site Plan toggle - they just never LEAD.
- A TEXT-BEARING page is a SLIDE: its whole-page render is a slide screenshot and is NEVER a hero. Only an image-only page (no text layer) may use its render, and only when the render is itself a photo.
- G-images now BLOCKS a non-photo hero (`classify_data_uri(hero) != "photo"`), signed off by `nonphoto_hero_ok: [ids]` in `placeholder_audit_ack.json` (mirrors `duplicate_photos_ok` / `placeholder_rate_ok`). LESSON: the independent reviewer had FLAGGED the bad heroes on the real run, but the gate only ADVISED, so they shipped - detection without a BLOCKING gate is worthless. Duplicate-hero floor lowered 3->2 (a single duplicated PAIR is wrong; the #22/#23 case slipped a >=3 rule).
- CBRE Spanish OPTION decks flatten each option to ONE slide carrying ~10 images (a photo + a site plan + a location map + icons), so the per-property page holds every kind at once - the tier rank is what keeps the photo leading on that page.

## LLM cross-source match adjudication (exit 10, 2026-06-19)
- The deterministic rule-based matcher (`match.same_property`) was NOT reverted (a past rule-based PHOTO matcher was); it is kept as the AUTO (confident) + FORBIDDEN (hard-blocker) tiers, the grey RECALL pre-filter, AND the OFFLINE FALLBACK. Only the genuinely ambiguous GREY-zone cross-source pairs are now LLM-adjudicated - the sanctioned "LLM judges equivalence, the coord/area/developer gates BLOCK, the coverage dedupe VERIFIES, Python clusters deterministically" pattern, identical in shape to photo-match (exit 9) + interpretation (exit 3).
- `pair_class(a,b)` -> auto|grey|forbidden|no. `run.py` enumerates grey pairs (`match.grey_pairs`, pure Python, recall-filtered to a handful), writes `work/match_candidates.json`, exits 10; the isolated sub-agent writes `work/match_decisions.json` ({pair_id:{verdict,reason}}, default 'different' when unsure); the re-run feeds it to `match.dedupe(records, decisions)` via `merge --match-decisions`. Verdict cached by an order-independent `pair_id` -> byte-deterministic; an uncovered/malformed/bad-verdict file re-emits + exit 10. Contract: `reference/matching.md`.
- BLOCKER-BEATS-LLM is by construction: `same_property` returns False for a forbidden pair BEFORE reading `decisions`, and the manifest only ever holds grey pairs. KEY REFACTOR LESSON for byte-identical backward-compat: classify AUTO **before** FORBIDDEN, and make `_cross_source_auto` reproduce the old fuzzy tail VERBATIM (incl. the pre-LLM matcher merging a same-key pair across a dev disagreement) - so FORBIDDEN only ever labels pairs the matcher would NOT have merged, a no-op offline and a veto only on the LLM. Proven by a DIFFERENTIAL fuzz of the verbatim old function vs the new `same_property(a,b,None)`: 0 mismatches across 3.1M pairs. The grey no-decision fallback returns False (a grey pair is never auto, so old code returned False too).
- Test gotcha: `merge._SRC_RESOLVE` memoises source paths by BARE FILENAME across the whole process, so a new full-spine test must use a UNIQUE input filename or it serves a later same-named test a stale deleted-temp-dir path (FileNotFoundError). No template change (still v16); manifest 48/37; battery green.
