# The gates

Same principles as `cbre-il-account-briefing`: **author != reviewer** (judgement -> isolated sub-agent; invariant -> script; orchestrator only adjudicates) and **shift-left** (validate data before the expensive build). Each script gate prints a scorecard fragment ending in one machine-read `STATUS: ALL-PASS|BLOCKED`. Each judgement reviewer writes `reviews/<gate>.md` ending in one `VERDICT: <green|amber|red>`. `final_gate.py` keys on both.

## Verdict semantics (machine-parsed - the contract every reviewer must follow)
- **`VERDICT: green`** - clean against this gate's rubric. Ships.
- **`VERDICT: amber`** - NON-BLOCKING observations only. Ships; the notes go to the Gaps Report. A reviewer must NOT use amber for a defect that meets the gate's "Blocks on" criteria - that is a red. (`final_gate.py` passes amber; it never blocks on it.)
- **`VERDICT: red`** - the gate's "Blocks on" criteria are met. Blocks the final gate until fixed and re-reviewed.
- The verdict is the **last line of the file**, alone, in exactly the form `VERDICT: <word>`. `final_gate.py` parses only line-initial `VERDICT` lines (prose like "the verdict would be red if…" never triggers), takes the last one, and **treats a missing or unrecognisable verdict as a BLOCK** - an empty or malformed review can never silently pass.
- **Findings must be machine-actionable.** Above the verdict, list each finding as one line:
  `- [HIGH|MED|LOW] property=<id|-> field=<field|-> issue=<one sentence> action=<one sentence>`
  so the orchestrator can apply fixes without interpreting prose. HIGH findings require a red verdict; MED follows the gate's "Blocks on" column; LOW is always amber-or-green material.

## Pre-build (Stage 4)
| Gate | Owner | Check | Blocks on |
|---|---|---|---|
| G-schema | `gate_runner.py validate-data` | schema valid; `*Val` numeric; rent display/numeric pair consistent; regionCode resolves; ids unique | any schema/consistency defect |
| G-selfcheck | `gate_runner.py self-check` | template has every `{{token}}` + the 3 markers; schema loads | doc/template/code drift |
| G-coverage | `gate_runner.py coverage` | every cluster produced records; no dup (park+city+dev+area); per-record core fill >= threshold or explicit `"tbd"` | silent thin record / dup |

> The dedupe check (no two cards share park+city+developer+warehouse-area) is the **VERIFIER for the cross-source match sub-agent** (exit 10, `reference/matching.md`): a wrong SPLIT - the sub-agent calling one property 'different' from its own twin - leaves two identical-tuple cards and BLOCKS here until merged, so an over-split is a visible, recoverable failure. The opposite error (a wrong over-MERGE) is prevented structurally upstream by `match.py`'s **forbidden** tier (a developer disagreement / >15% size conflict / same-source differing-area pair never merges, even on an LLM 'same' verdict). Together: coord/area/developer = the blocker, the coverage dedupe = the verifier. `trace-coverage` still requires every merged field to trace to a source.
| G-trace-coverage | `gate_runner.py trace-coverage` | (mechanical backstop) every non-sentinel, source-able property field has a `source_ledger.csv` row whose source_type != `gap` | a populated field with NO ledger row (possible fabrication) |
| G-images | `gate_runner.py images` + isolated Sonnet reviewer | script: every `photo` is a valid data URI; unmatched assets listed; **a PLACEHOLDER whose source page held discarded image candidates BLOCKS until reviewed** (merge dumps every candidate to `render/placeholder_audit/`; "no usable image" must be a reviewed conclusion, never a silent default); **>= 3 properties sharing ONE identical hero BLOCKS** (a near-certain harvest failure; sign-off key `"duplicate_photos_ok": ["<hash12>"]`); **a longlist >= 4 properties that is >= 50% placeholders BLOCKS** even with no discarded candidates (sign-off key `"placeholder_rate_ok": true`). Reviewer judges **one exhaustive `contact_sheet.py` montage of ALL N photos** (tiles render at 480px; if a tile is ambiguous - logo vs real photo - open that property's full `photo` data URI in `canonical.json` before flagging, rather than calling it off the thumbnail) PLUS the `placeholder_audit.png` discard-pile montage: for each placeholder, either name the usable photo/plan to rescue or sign off that none exists; the orchestrator records the verdict in `<work>/placeholder_audit_ack.json` (`{"confirmed": ["<id>", ...]}` plus the keys above) and applies any rescue | mis-matched / invalid image / an unreviewed placeholder with discarded candidates / an unreviewed duplicate-hero group or placeholder-heavy longlist |
| G-trace | isolated Opus reviewer | sampled fields trace to their cited locator AND the value matches the source (existence is covered mechanically by G-trace-coverage; **a `description` tagged `(brochure description, text interpretation)` is an LLM verbatim pick from the photo-matched deck, already verified in-code by a deterministic quote-check before it could enter the record, so give it the same re-read scrutiny as any interpreted source - the plain `(brochure description)` tag is the deterministic font-size heuristic**). **DECISION-CORRECTNESS (not just value-at-cell):** (a) **tracker basis** - for any `warehouseArea`/`plotArea`/`officeArea`/`*RentVal` whose `source_ledger.csv` `source_locator` names a basis (`GIA`/`GEA`/`GLA`/`warehouse`) or a derivation (`warehouse = GIA X - office Y`), re-derive the basis/unit from the sample rows in `work/yield_report.md` / the tracker `work/extract/*_map.json` and confirm it is right for those values (a column called `warehouseArea sq m` carrying 172,867 is almost certainly sq ft; a `GIA` basis with no office subtracted is suspect). (b) **cross-source match** - for any property whose `meta.conflicts` shows a cross-source merge, re-read BOTH pre-merge source records and decide for yourself whether they are one physical property. (c) **read C's machine diffs** - treat any `semantic_disagreement` line in the yield report (two independent map passes disagreed) or any `match disagreement` line in `meta.conflicts` (the blind match verifier disagreed) as a **RED-unless-resolved** prompt: re-derive the basis/verdict yourself; AMBER + a finding line if it is a defensible difference the broker should confirm, RED only if the chosen decision produced an impossible number or a wrongly-fused record. | any untraceable / mis-stated field (struck to `"tbd"`); a decision the reviewer re-derives as wrong AND that shipped an impossible value / a fused record |
| G-honesty | isolated Opus reviewer | every `"tbd"`/`"—"`/`null` genuinely unknown; no invented number; conflicts surfaced; footer disclaimer intact. **DECISION-CORRECTNESS:** confirm every decision conflict is genuinely SURFACED in the Gaps Report, not silently resolved - in particular that each C `semantic_disagreement` (map/basis) line and each `match disagreement` line in `meta.conflicts` actually reaches the broker (a swallowed disagreement is a dishonesty defect, same class as a fabricated value). A confidently-wrong interpreted decision that ships a fabricated attribution (a basis that makes the shipped number impossible, a `same` verdict that fused two genuinely different addresses into one blended record) is a HIGH/RED. | any fabricated value / a swallowed decision disagreement / any HIGH |
| G-enrichment | `gate_runner.py enrichment` | (the layer that had no gate) cross-region **duplicate figure** smell (an unemployment/GDP figure copied from a neighbour); every stated figure has an `*AsOf` + `sources`; units/range sanity; **POIs/distances not empty** (the empty-POIs bug) - unless flagged DEGRADED/offline | copied figure / unsourced figure / empty POIs |
| G-enrich | isolated Sonnet reviewer (only when `regions` enrichment ran) | the default workforce snapshot is now supplied wholesale by the bundled Oxford Economics dataset, so verify: each region resolved to the **right province** (a broad/bilingual label must not bind a neighbour - check the name against the cited province; **a label resolved by the interpretation sub-agent (region_labels job -> work/extract/region_labels.json) gets the SAME right-province scrutiny as a dataset match** - re-verified in-code via `_dataset_region` and overridden by the coordinate point-in-polygon bind when coords exist, but a coord-less LLM resolution onto the wrong neighbouring province is still a blocking wrong-province bind here), the bundled **citation is present** on every profile, and the **derived logistics-employment share is plausible** (transport & storage employment < labour force, share not absurd); AND for any RESEARCHER-OVERRIDDEN figure (a region the dataset does not carry, or an extra such as employmentRate/gdpPpsEu) the full source rules still apply - it traces to the **right province** in its cited source, the citation is current, **unemployment meets the recency FLOOR: current year preferred, current-1 minimum; Jan-May only, current-2 passes solely with a `recencyNote` documenting the failed current-1 search** (the mechanical gate enforces the same rule); GDP/population may lag ~2 years; **SOURCE QUALITY: a researched figure cites the most authoritative source available - convenience sourcing (an aggregator, or a pre-approved/easy source when a more authoritative or more current one exists) is a blocking defect, same class as fabrication** | wrong-province bind / missing dataset citation / implausible derived share / a researched figure wrong vs source / stale citation / unemployment below the floor / a better source available but a more convenient one cited |

Freeze `canonical.json` only when `gate1_scorecard.md` reads `STATUS: ALL-PASS` (`run.py` does this automatically at ALL-PASS and removes a stale snapshot on a blocked scorecard).

## Post-build (Stage 6)
| Gate | Owner | Check | Blocks on |
|---|---|---|---|
| G-html | `gate_runner.py validate-html` | delivered HTML == `render(canonical)` byte-for-byte; 3 blocks round-trip; chrome sha == VERSION | chrome drift / parse failure |
| G-reconcile | `gate_runner.py reconcile` | every id HTML <-> canonical; hero KPI strip matches the data | new/dropped record / KPI mismatch |

## Independence + model/effort
Model matched to risk: **G-honesty and G-trace = Opus/High** (a wrong answer ships fabricated data - the worst failure); **G-images, G-visual and G-enrich = Sonnet** (image-match, layout, and - now the workforce snapshot is dataset-sourced rather than researched - the lighter enrichment check are easier, much faster judgements). **Reserve Opus for those two fabrication-risk reviewers only; run the orchestration itself and the gate re-checks on Sonnet** (the run's biggest token sink is the orchestration, and most of it - intake adjudication, photo-match, verdict triage - does not need Opus). Mechanical scripts gate first so judgement agents never spend tokens on data that fails the free checks. **Scale reviewer EFFORT to risk (this is about depth, never about independence).** When the artefact is clean on its face - a token-clean build, a workforce block that is 100% bundled-dataset-sourced with current dates and a plausible derived share - the reviewer confirms quickly and returns green/amber; it does not re-derive from scratch. The instant it sees an anomaly (a figure that smells copied, a placeholder with discarded candidates, a value absent from the source), it switches to full scrutiny. A fast confirm on genuinely clean data is correct, not a shortcut; a thorough hunt the moment something is off is mandatory. (G-enrich especially: post-v11 the snapshot is dataset-sourced, so the common case is a fast confirm, the exception is a researcher-overridden figure that gets the full source/recency check.) Reviewers never edit the artefact - they return a verdict; the orchestrator applies the fix and re-runs with a **fresh** reviewer. A defect fixed in the template (not the data) means bumping the template version (`reference/template-contract.md`).

## Reviewer dispatch contract (context independence) - MANDATORY
Each judgement reviewer must be **fresh-context and blind to the orchestrator's view**. This is non-negotiable: a reviewer that inherits the orchestrator's reasoning is not independent, and the gate is then theatre.

1. **Fresh context.** Dispatch each reviewer as a separate `Agent` sub-agent (one gate = one agent). It inherits NONE of the orchestrator's conversation; it starts from an empty window. Never reuse a single agent across gates (judging honesty and visual in one head cross-contaminates).
2. **Blind dispatch.** The prompt the orchestrator hands the reviewer contains ONLY:
   - the **path(s)** to the artefact under judgement (`canonical.json`, the built `.html`, or `render/*.png`), which the reviewer **opens and reads itself** - never paste the orchestrator's summary of it. For the DATA reviewers (G-honesty / G-trace / G-enrich) hand them `work/canonical_review.json` - a photo-stripped twin regenerated on EVERY freeze, including a manual re-freeze after an out-of-band data fix, so a re-review can never read a stale twin (the base64 heroes are ~99% of the multi-MB `canonical.json` and only burn the reviewer's context); `canonical.json` stays authoritative and the contact sheet is the image aid for G-images;
   - the **gate rubric** (the checklist from this file / `reference/visual-qa.md`) and, where relevant, `templates/canonical.schema.json` or the `assets/reference_screenshots/` baseline.
   - **for the decision-correctness checks (G-trace / G-honesty), C's two machine diffs and their evidence**, which the DATA reviewer opens read-only itself: the tracker `work/extract/*_map.json` + the matching `work/extract/*_mapcheck.json` (the second blind map; the `semantic_disagreements` are surfaced in `work/yield_report.md`), and `work/match_decisions.json` + `work/match_verify.json` (the author and the blind verifier verdicts; the `match disagreement` lines ride `meta.conflicts`), alongside `source_ledger.csv` (the basis assumption is in `source_locator`) and the pre-merge `work/extract/*.json` source records. These are the **decision's own audit trail and a machine SECOND-PASS DIFF, not the orchestrator's view** - the reviewer SEES C's diff but still re-derives the basis/verdict from the raw rows/records itself and forms its OWN verdict; the MUST-NOT-CONTAIN list below is unchanged.
   It MUST NOT contain: the orchestrator's own findings or opinion, an "expected" answer, "I believe this is correct / fine", which fields the orchestrator thinks are weak, or what any previous reviewer concluded. The reviewer re-derives its judgement from the artefact alone.
3. **Verdict only, written by the reviewer.** The reviewer writes `reviews/<gate>.md` ending in one `VERDICT: <green|amber|red>` line. The orchestrator never writes or edits a verdict, and never self-certifies a gate over data it authored.
4. **Fresh on every re-run, SCOPED to what changed.** After a fix, re-dispatch a NEW reviewer with no memory of prior rounds - do not say "you previously flagged X" or "re-check your earlier note". Re-review is **scoped**: re-freeze the artefact, then re-dispatch ONLY the gate(s) that returned red - not the whole batch. Verdicts from the other gates remain standing for a targeted data fix. Re-dispatch the FULL batch only when the fix added/removed properties or spans several gates' domains (e.g. replacing photos AND rewriting commercials). Bounded loops (~3); if a gate still will not clear, escalate to the Gaps Report or the user rather than loosening the rubric.
5. **DEGRADED mode is labelled.** If the skill is run in plain chat with no sub-agent dispatch available, the "reviewer" is the same context as the author: independence is lost. Say so plainly, mark the run DEGRADED, and treat the verdicts as self-review, not independent QA. This applies with FULL force to the decision-correctness checks: without dispatch, C's second map / match passes cannot run blind (no `*_mapcheck.json` / `match_verify.json`) and the G-trace/G-honesty decision re-derivation is the author re-reading its own work - label it self-review explicitly, never independent.
6. **Parallel dispatch (faster, and MORE independent).** Dispatch a phase's reviewers **concurrently** - all of a phase's `Agent` calls in a single message - not one after another. This is strictly safe and actually strengthens independence: when they fire together no verdict exists yet, so none can be biased by another's conclusion. Two hard rules make parallelism quality-neutral:
   - **Still one gate = one agent.** Never collapse gates into a single "review all dimensions" agent to parallelise - that destroys independence. Concurrency means N separate fresh agents at once, each with its own rubric and its own `reviews/<gate>.md`.
   - **Freeze the artefact around the window.** `run.py` freezes `canonical.json` AUTOMATICALLY when the pre-build mechanical scorecard is ALL-PASS (and deletes a stale snapshot when it is blocked), so the snapshot exists before you dispatch reviewers without a manual step; re-running `gate_runner.py freeze <canonical.json>` by hand is only needed after an out-of-band edit - and that re-freeze now ALSO regenerates the photo-stripped `canonical_review.json` twin, so re-dispatched DATA reviewers can never read pre-fix data (this closes the wasted duplicate-review-round). After collecting verdicts, `gate_runner.py freeze <canonical.json> --check` must pass - this proves every concurrent reviewer judged byte-identical bytes and that no edit slipped in mid-review. `final_gate.py` re-checks this (skipped only in `--no-reviews` DEGRADED mode). Pre-build reviewers (G-honesty, G-trace, G-images, **and G-enrich when `regions` were enriched**) run in one concurrent batch against the frozen `canonical.json`; G-visual runs post-build against the built HTML. (`final_gate.py` requires a `G-enrich.md` verdict whenever regions ran, so it must be dispatched in that batch - omitting it deadlocks the final gate.)
