# Data engine - extraction, matching, merge, images

## Extraction (Stage 1)
Each extractor emits candidate partial-property records (canonical field names) plus per-field provenance, and writes unknowns as `"tbd"`/`null`, never guesses.

- **PDF** (`extract_pdf.py`, the preferred FIELD source). Two brochure formats are handled: a numbered Table-Of-Contents on page 1 (properties from page 2), or no TOC (page 1 is already a property). Parsing is **label-anchored**: a regex alternation of known labels (with `\s+` so a label may wrap across lines) locates each label; the text between one label and the next is its value. Area fields take the FIRST plausible (>=1000) number (a prefix like "BOR 10:" is filtered by the >=1000 floor; "the largest" used to turn "39 471 (expandable to 80 000)" into 80000); a RANGE keeps its honest text (plotArea as the value, a warehouseArea range goes to divisibleFrom - warehouseArea is schema-numeric). Rents REQUIRE currency/unit context (€/EUR/£ or /m²-style) or a figure inside the 1.5-500 EUR/m²/yr plausibility band - a bare big number under the short "Warehouse" label is a size, not a rent; monthly quotes (regex-matched, spaces tolerated: "/ sq m / month", "/ m2 / mes") are annualised x12 and render as `€<val> / sq m / year`, and an implausible figure ships as text only, never as a numeric. A prose DESCRIPTION block is rescued out of the value stream; the page footer is stripped. Coordinates come from a `lat,lng` regex over the page text, or - stronger, and fully offline - from the page's **maps hyperlink** ("click for location" links carry `?q=lat,lng`): FIRST-PARTY coordinates pinned by the brochure author, recorded with locator `page N (map link)`, no geocoder needed. Motorway is mined from the description ("next to the D5 motorway") when there is no explicit label. Park name = the TOC entry, else developer + city (the reference's "WING Sosket" style). Label map in the `LABELS` list - extend it for new brochure vocabularies.
- **PPTX** (`extract_pptx.py`, the preferred IMAGE source). One property per content slide; reuses the PDF label machinery on the slide text (messier than PDF, hence fallback for fields). Harvests the most PHOTOGRAPHIC slide picture as the hero (`photographic_score`, same scorer as the PDF path - not the largest, so a logo/plan cannot win by area). A slide-sourced record WITHOUT a pre-embedded photo (e.g. a vision transcription of a deck, `source_type: "pptx"` + `page_no` = slide index) harvests at merge via `images.slide_hero_and_plan`: LibreOffice (when installed) renders the deck to PDF and the FULL page ladder runs on the converted page (vector slides included), else the slide's embedded pictures are scored directly; a slide that still yields nothing feeds the placeholder audit exactly like a PDF page.
- **XLSX** (`extract_xlsx.py`). Auto-detects a tracker (header row with >=3 known columns -> records) vs a questionnaire (label/value pairs). The column->field DECISION is the only ambiguous part, and it is now LLM-adjudicated: an isolated tracker-interpretation sub-agent maps the WHOLE header (`run.py` offers it via the same exit-3 manifest as the brochure decks - a `kind:"tracker"` job carrying the raw headers + a few sample rows; the sub-agent returns a MAP, never records, never a cell value; see `reference/interpretation.md` "Tracker mode"). The hardcoded `COLUMN_MAP` dictionary is no longer the sole mapper but stays LOAD-BEARING in three roles: the OFFLINE FALLBACK (when no map exists, `column_map=None`, the dictionary path runs verbatim and byte-identical - this is what the offline evals exercise), a HARD NEGATIVE-table VETO on the LLM map (a column whose header is a derived/penalty figure - `Rent free (months)`, `Size Unit` - can never bind to a data field, even if the LLM says so), a backfill of any column the LLM leaves unbound, and a logged cross-check of confident disagreements. Either way **Python parses every number** with the same arithmetic (the LLM only NAMES units/basis), so the engine is the confident-common-case mapper + verifier and the LLM adjudicates the long tail (a continental sheet the dictionary maps thin, a 75-column tracker that mapped ~7). The map is WIDE (~28 fields: eaves/clear height, office content, floor loading, doors, power, parking, EPC/BREEAM, site area, combined or separate lat/lng columns split into exact first-party coordinates); collisions resolve by alias priority + header coverage, with losing columns kept as per-row fallbacks (a "Promoter" column fills `developer` when the "Developer" cell is empty - a WITHIN-field fallback). **Landlord and developer are DISTINCT fields, never conflated**: a "Landlord"/"Owner"/"Asset manager"/"Freeholder" column maps to its own `landlord` field (the owner / asset manager of an existing building) and NEVER backfills `developer`. The old conflation (landlord/owner/asset-manager listed as `developer` synonyms + a Landlord column filling an empty Developer cell) shipped a landlord as the developer, and a landlord-vs-developer difference then split one property into two cards; that backfill is removed. A "Landlord Verified" column is kept off the field by a NEGATIVE-table regex. "Developer beats Landlord" for the developer field is automatic now (Landlord no longer competes for it). Units live in the HEADER (or the LLM's explicit hint) and the SOURCE CONVENTION IS KEPT: "(sq ft)" areas stay sq ft (`areaUnit`), "£ per sq ft" rents ship as £/sq ft/yr (`rentUnit`, own plausibility band 0.5-60), "(m)"/"(kN/sq m)"/"(KVA)" suffix bare numerics, acres convert to sq ft (prov-noted). A sheet mapping >=8 fields is a RICH tracker (`__meta.tracker_rich`) with merge spec authority; every sheet emits a `header_report` (populated vs mapped columns + `map_source`/`llm_vetoes`/`llm_dictionary_disagreements` when LLM-mapped) and run.py writes `<work>/yield_report.md` when a rich sheet parses thin - the Gaps Report surfaces it. The LLM map is cached at `work/extract/<slug>_<hash8>_map.json` keyed by a STRUCTURE hash (headers + sample rows), so identical structure -> identical map -> byte-identical records + ledger (resume-safe); a `.SKIP` sentinel keeps the dictionary for an offline / no-LLM run. **The column->field MAPPING/BASIS is checked by an INDEPENDENT blind SECOND PASS** (not just the same-model reviewers): `run.py` emits a sibling `kind:"tracker_verify"` job on the same exit-3 manifest, a SEPARATE fresh agent re-derives the SAME map blind into `work/extract/<slug>_<hash8>_mapcheck.json` (same `input_hash`), and `extract_xlsx.diff_tracker_maps` diffs the two in pure Python -> a `semantic_disagreements` `header_report` entry -> the yield report / Gaps Report. ADVISORY only - the primary map still drives the parse, never auto-rejected; absent mapcheck -> empty diff -> byte-identical (`reference/interpretation.md` "Verification pass"). A tracker's records flow through merge; a questionnaire's pairs are carried into `meta.requirements` by the spine (`merge --requirements`, fed from `run.py`) for the orchestrator's size-slider default + hard-requirement flags. Excel is authoritative for commercials when present.
- **Images** (`extract_image.py`, ORCHESTRATOR-driven - NOT in the `run.py` spine). Property-named files -> bindable assets (`{kind, tokens, data_uri}`); filename tokens are match hints; HEIC supported; "plan/floor/layout" filenames are tagged `kind=plan`. The orchestrator binds each asset to a property's `photo`/`plan` field by token match (the spine harvests heroes from the brochures themselves and never invokes this reader).
- **Email** (Outlook MCP, agent-driven). The isolated sub-agent calls `outlook_email_search` with **`folderName`** = the broker's named Outlook mail folder from `inputs.emails.outlook_folder` (Inbox/Sent/Archive resolve directly; any other name is looked up by enumerating the mailbox), omitting it to search all folders; add `mailboxOwnerEmail` (`inputs.emails.mailbox`) for a shared/delegated mailbox, plus a `query`/date scope, then reads matches with `read_resource`. `extract_email.py` is the `.msg`/`.eml` Windows-folder fallback for teammates without the Outlook MCP. Newest email wins commercials. **Map links:** an offer email often pastes a Google/Apple/OSM maps link or a bare `lat,lng` pair; the sub-agent copies that raw string into the record's `__meta.map_candidates` (a list of strings) and never sets `lat`/`lng` itself - the shared resolver parses it (see "Map-link coordinate resolver" below), so a first-party pin beats the town-centre geocode with no model-invented coordinate.

**Map-link coordinate resolver (`extract_pdf.backfill_link_coords`, cross-input).** One deterministic pass, run post-merge / pre-geocode over records from EVERY source, fills `lat`/`lng` + `mapLink` from a first-party map link wherever one hides, then leaves anything unresolved a gap. The URL/coordinate grammar lives in the shared `coords.py` (`MAPS_URI` + `LINK_LL` destination-only keys + a strict `PLAIN_LL` bare pair with >=3-decimal + bounds guards, so an area magnitude or a low-precision ratio never false-positives). Three passes: **(A)** stashed `__meta.map_candidates` strings (Excel cells - stashed per-row by `extract_xlsx`; email bodies - stashed by the sub-agent above; the key is consumed and deleted so it never reaches `canonical.json`); **(B)** PDF re-open - the page's hyperlink ANNOTATIONS then the visible page TEXT; **(C)** PPTX re-open - the slide text + its shape/run hyperlink addresses. Precedence is unchanged (a dedicated `lat`/`lng` column and an existing pin win; a record with a numeric coord AND a `mapLink` is skipped) and it NEVER invents - an unresolvable short link (`goo.gl`, `maps.app.goo.gl`) ships as `mapLink` with the coord left for the web-enrich round. Degrades to a no-op for any pass whose engine (`fitz` / `python-pptx`) or source file is absent.

## Matching (`match.py`)
Key = normalised City + Developer + Park (diacritics stripped, legal suffixes removed, developer aliases like CTPark->CTP). Two records from **different** sources are the same property when the key matches >= 88 (rapidfuzz token-set) - unless exactly one record's park is empty, in which case the areas must also agree (token-set scores a missing park as a subset match at 100, which would merge every same-developer option in a city). Cross-source pairs ALSO match by **coordinate proximity**: pins <= 300 m apart with no developer disagreement (unknown = absent, never a disagreement) and no material size conflict are one property - an unknown city defeats every text key, so a vision record carrying the real city never matched its city-less deterministic twin without this. Distinct phases sharing one pin stay separate via the same +/-15% area rule. Records from the same brochure stay distinct (two pages with the same park name are distinct buildings/phases, distinguished by warehouse area) - EXCEPT a true restatement of one unit (identical key and identical/absent area, e.g. a summary-table row plus that unit's detail page), which merges so the coverage gate's duplicate check cannot block the skill's own output.

**The equivalence DECISION is tiered (`pair_class`), and the genuinely ambiguous middle is LLM-adjudicated** (the same "LLM judges, Python verifies, gates block" discipline as the brochure and tracker sub-agents). Every record PAIR falls into one of four tiers: **auto** (today's confident TRUE paths above - merged deterministically, the LLM is never asked the easy questions), **forbidden** (a HARD blocker - a >15% size conflict or a same-source differing-area phase - that can NEVER merge even on an LLM 'same' verdict, because `same_property` returns False before consulting the decision; a developer DISAGREEMENT is NO LONGER forbidden - landlord and developer are distinct fields now, so a genuine developer-name difference is a grey signal the LLM adjudicates, not a veto), **grey** (cross-source, not forbidden, not auto, but clears a RECALL pre-filter: same normalised city OR within ~2 km OR sharing >= 1 distinctive park token OR a borderline fuzzy key in [70, 88) - this includes a developer-disagreement pair; the coord-net AUTO path still requires developer agreement, so a disagreement is never auto-merged), and **no** (definitely distinct). Only **grey** pairs are shown to the LLM. `run.py` enumerates them with `match.grey_pairs` (pure Python, recall-pre-filtered so it is a handful, typically 0-5, not an O(n^2) LLM call), writes `work/match_candidates.json` and exits 10; an isolated sub-agent writes `work/match_decisions.json` ({pair_id: {verdict, reason}}, default 'different' when unsure); the re-run feeds it to `match.dedupe(records, decisions)` via `merge --match-decisions`. Each pair carries a stable order-independent `pair_id` (a hash of the two records' sorted match_key+area), so the verdict is cached and clustering is byte-deterministic; a NEW grey pair an input edit introduces re-emits the manifest + exit 10 rather than silently guessing. **The deterministic matcher is NOT deleted** - it is the auto/forbidden tiers, the grey RECALL pre-filter, AND the OFFLINE FALLBACK: with no decisions file (`decisions=None`, every offline eval) `same_property`/`dedupe` are byte-identical to the pre-LLM matcher. **The grey MATCH verdict is checked by an INDEPENDENT blind SECOND PASS** (not just the same-model reviewers): `match_candidates.json` also carries a `verify_pairs` array (the same grey pairs, both records, NOT the verdict); a SEPARATE fresh agent re-judges blind into `work/match_verify.json`; `run.py` diffs the two verdicts per `pair_id` in pure Python and folds any disagreement into `meta.conflicts` -> the Gaps "Source conflicts" section. ADVISORY only - the matching pass's verdict still drives clustering, never flipped; absent `match_verify.json` -> empty diff -> byte-identical. See `reference/matching.md` for the sub-agent contract.

## Merge (`merge.py`)
Cluster -> one property by field-class precedence: **commercials** (rent/terms/incentives/land) newest email > excel > brochure; **specs/geo/everything else** brochure (pdf > pptx) > excel > email. The static brochure order is **quality-aware** (`compute_file_quality`, the same probe run.py routes files to vision with): a brochure whose records MOSTLY parsed poorly loses every precedence contest to a cleaner twin - "PDF preferred for fields" holds only while the PDF parse is actually reliable, so a print-export PDF with a flattened text layer can no longer outrank its clean PPTX twin on rank alone. Non-brochure sources (xlsx/email) are never demoted - their records are legitimately sparse. A **RICH tracker** (`__meta.tracker_rich`, >=8 mapped columns) additionally LEADS the structured spec fields and coordinates (`TRACKER_AUTHORITATIVE`: areas, clear height, floor load, doors, power, parking, BREEAM, status, early access, lat/lng) - curated internal data beats brochure prose for measured values; naming and narrative stay brochure-first. Merge also owns the **dataset unit convention**: the dominant `areaUnit` is stamped on every property (minority-unit areas convert arithmetically, prov-noted), `meta.units` records both conventions, and currency is NEVER converted (a lone €/m² rent in a £/sq ft dataset keeps its own honest unit and sits out the hero rent range). Assigns stable ids, attaches a compressed base64 hero by **image** precedence (a PPTX slide picture if one was extracted - the preferred image source - else a raster of the brochure PDF page, else a CBRE placeholder), fills every chrome-read key with its sentinel and coerces honest numerics in string-typed fields (`canonicalize` -> `loadingDocks: 12` becomes `"12"` so it can't fail the schema), and writes `canonical.json` + `source_ledger.csv`. **Rent contract:** `warehouseRentVal` is the ANNUAL EUR/m²/year figure shown in `warehouseRent`; a monthly quote (`/mes`, `/month`, …) is annualised x12, with the conversion recorded in the ledger. Note the hero's image precedence is the inverse of the field precedence: PDF wins fields, PPTX wins pictures.

**Genuine cross-source VALUE conflicts are LLM-adjudicated on the SAME exit-10 dispatch as the match grey pairs** (the same "LLM judges, Python verifies, gates block" discipline). When two+ clustered sources hold DIFFERENT non-unknown values for one field, `run.py` enumerates it with `merge.conflict_candidates` (pure Python, recall-pre-filtered to genuine disagreements, typically 0-3) and folds it into `work/match_candidates.json` as a `field_conflicts` array alongside the `pairs`; an isolated sub-agent writes `work/field_decisions.json` ({conflict_id: {pick, reason}}); the re-run feeds it to `merge --field-decisions`. **The fixed precedence is the DEFAULT.** A pick OVERRIDES it ONLY when it SELECTS one of the given candidate values (never a free/invented value) AND that value PASSES the field's deterministic plausibility gate (rent band `normalize.rent_unit_band` / area > 0 / coordinate bounds); a failing or absent pick keeps precedence, noted in `meta.conflicts`. Each conflict carries a stable order-independent `conflict_id` (a hash of the merged match_key + field + the SORTED disagreeing values), so the pick is cached and merge is byte-deterministic offline; a NEW conflict an input edit introduces re-emits the manifest + exit 10 rather than silently guessing. **The fixed precedence is NOT deleted** - it is the default the LLM only overrides with a high-confidence gate-passing pick AND the OFFLINE FALLBACK: with no decisions file (`FIELD_DECISIONS={}`, every offline eval) `merge_cluster` falls straight through the precedence ladder, byte-identical to the pre-LLM merge. Every override + rationale (and every gate veto) reaches the Gaps Report (auditable, never silent). See `reference/matching.md` for the sub-agent contract.

## Images (`images.py`)
Per-image byte budget (~110 KB): resize to <=1280 px and step JPEG quality down until under budget; base64 data URI. Keeps the assembled file a few MB, not the reference's 11 MB.

**Hero selection is by `photographic_score`, not by area - in FOUR engine-agnostic tiers.** A pure-pixel-statistics score (colourfulness + non-white fraction + aspect sanity; no brand/colour/region hardcoded) judges every candidate; selection is **page-scoped** (keyed on `__meta.page_no` - vision records must carry it VERBATIM from the manifest; see `reference/vision-fallback.md`), so a multi-property brochure binds each property's hero to its own page.
- **Tier A - embedded image:** the most photographic decodable embedded image at/above the 320x200 floor. At/above `MODEST_PHOTO` (6.0 - plans/maps/logos score 0-5, real photos 15-37) it wins OUTRIGHT; the page raster can never beat an actual photo (it used to, shipping cluttered full-page tiles).
- **Tier B - bbox crop:** when the active PDF backend cannot DECODE the image streams, the most photographic image REGION is cropped out of the page render using pdfplumber's bboxes (pure geometry - works on every backend).
- **Tier C - page render:** whichever of {best sub-modest embedded, whole-page raster} scores higher (a standalone logo/plan loses to the page). In a renderer-less sandbox (pdfplumber-only) this tier degrades to the best embedded image.
- **Tier D - placeholder**, recorded as an honest gap row.

**SITE PLANS are harvested too** (the modal's Photo/Site-Plan toggle reads `p.plan`): the page's best non-hero image region with the PLAN SIGNATURE - a balanced mix of white paper and drawn ink (white fraction 0.15-0.90, balance x area ranked). The hyperlinked location map is excluded by its maps-service link sitting on/under the box; boilerplate logos/legends are excluded by image-OBJECT identity repeating on >=3 pages (never by position - templated brochures place content at identical positions every page). Combination rules: photo found -> photo is the hero, plan fills the plan slot (or stays absent); plan-only page -> the plan IS the hero and fills the plan slot; a record-level `plan` data URI (orchestrator-bound standalone file) outranks the page plan.
Engines: PyMuPDF preferred; `fitz_shim` falls back to pypdfium2, and (new) to pdfplumber-ONLY - where text and embedded JPEG/JPEG2000 photos still extract (decoded straight from the PDF streams) and only page rendering is unavailable. Heroes are memoised on disk per (source, page, budget).

## Known quality notes
- Coordinates absent from a brochure are filled by `enrich.py --geocode` (Nominatim, cached, `coordsApprox: true`), which also **reverse-geocodes the country code** and fills an unknown (`??`) country - so any geography works without a region-specific city index. A compound city string that geocoding cannot resolve is an honest Gaps-Report line. Caches (geocode/regions/POI) are seeded read-only in `reference/` but written to the **work dir** (`--cache-dir`), so a read-only install can still cache and be pre-filled.
- Motorway is `tbd` where the prose does not mention one - genuine, not a defect.
- Extend `assets/label_ledger.json` (multilingual brochure labels - the PRIMARY extension point; EN/ES/FR/DE/IT/NL/PT/PL), the in-code `LABELS` (extract_pdf English baseline), `COLUMN_MAP` (extract_xlsx) and `DEV_ALIASES` (match) as new landlords / languages / templates appear; record the addition in `reference/memory.md`. A page counts as a property page on `>=2` recognised labels in ANY language. Own-line "Label / value" spec sheets are parsed directly; if a deck has NO own-line labels on any page, the extractor falls back to same-line "Label value" / "Label: value" parsing, tolerating a leading bullet (`- Altura libre: 11,50`), in any language (document-level switch, so spec-sheet decks are byte-identical). An inline page that yields only peripheral specs (clear height, docks) with **no core field** (city / developer / warehouse area / rent) is not emitted as a record, so a narrative/table deck (core data in the header + prose + a positional table) falls through to the **vision fallback** along with image/vector-only decks (no text layer at all): `vision_prep.py` rasterises the pages into `work/vision/manifest.json` and an isolated vision sub-agent transcribes them into `extract/<region>_vision.json` (records reuse the brochure's `source_type` with a `(vision transcription)` locator). See `reference/vision-fallback.md`. Rent quoted per month (/mes, /mois, /Monat, ...) is annualised x12 with the conversion recorded in the ledger.
